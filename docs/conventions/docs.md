# 문서 컨벤션

문서는 구현을 위한 합의 기록입니다. 단순 설명보다 “무엇을 기준으로 개발할지”와 “아직 무엇을 결정하지 않았는지”를 분명히 남기는 것을 우선합니다.

## 구조

- 기획 문서는 배경, 목표, 범위를 설명합니다.
- 설계 문서는 선택지, 결정 이유, 영향 범위를 설명합니다.
- API 문서는 OpenAPI와 사람이 읽는 설명을 함께 유지합니다.
- 운영 문서는 실제 배포/복구 절차를 기준으로 작성합니다.
- 회의 문서는 확정 전 선택지와 질문을 모읍니다.
- 컨벤션 문서는 구현 중 반복 판단을 줄이는 규칙을 모읍니다.

## 문서 상태

확정 여부를 문서 안에서 명확히 표현합니다.

| 상태 | 의미 |
| --- | --- |
| `Draft` | 초안. 구현 기준으로 쓰기 전 검토 필요 |
| `Proposed` | 기본 후보. 반대가 없으면 구현 기준으로 사용 가능 |
| `Open` | 회의 필요. 구현에 반영하지 않음 |
| `Accepted` | 결정됨. 코드/스키마/API 기준으로 사용 |
| `Deprecated` | 더 이상 신규 구현 기준으로 사용하지 않음 |

회의해야 할 내용은 `meetings/`에 두고, 결정되면 [의사결정 기록](../decisions.md)과 관련 설계 문서에 반영합니다.

## 작성 규칙

- 공개 가능한 정보만 작성합니다.
- 확정되지 않은 내용은 `초안`, `Proposed`, `Open` 등으로 표시합니다.
- 로컬 경로, 개인 이름, 비밀값, 실제 운영 credential은 쓰지 않습니다.
- 코드나 SQL 예시는 복사해서 사용할 수 있게 완전한 형태로 작성합니다.
- “나중에 정리”보다 현재 아는 범위와 남은 질문을 함께 남깁니다.
- 레거시 기준을 적을 때는 현재 기준인지 참고 자료인지 구분합니다.
- API/DB/운영 변경은 영향 범위를 같이 적습니다.

## 링크와 파일 참조

- 같은 문서 사이트 내부 링크는 상대 경로를 사용합니다.
- 문서명은 사용자가 찾는 주제 중심으로 짓습니다.
- 제목은 짧고 검색 가능한 단어를 사용합니다.
- 이미지나 다이어그램은 설명 없이 혼자 떠 있지 않게 앞뒤 문맥을 둡니다.

## 코드 블록

- shell 명령은 `bash` 또는 `text` info string을 사용합니다.
- Java 예시는 `java`, SQL 예시는 `sql`, JSON 예시는 `json`을 사용합니다.
- secret 값은 예시에서도 실제처럼 보이는 값을 쓰지 않습니다.

예:

```text
SPRING_DATASOURCE_PASSWORD=...
JWT_SECRET=...
```

## ADR 작성 기준

ADR에는 다음을 포함합니다.

- 상태: `Proposed`, `Accepted`, `Deprecated`
- 배경
- 결정
- 대안
- 영향 범위
- 후속 작업

결정이 바뀌면 기존 결정을 조용히 수정하지 않고 새 ADR 또는 변경 이력을 남깁니다.

## 문서 변경 기준

다음 변경은 문서 업데이트가 필요합니다.

- DB schema 변경
- API request/response 변경
- 인증/권한 정책 변경
- 배포 방식 변경
- 운영 runbook 변경
- 게임 시스템 정책 변경
- executor/job/worker 실행 경계 변경
- Redis, S3, mail 같은 외부 의존성 추가
- 보안/로그 정책 변경

## 정합성 점검 체크리스트

문서를 크게 수정하거나 구현을 시작하기 전에는 아래를 확인합니다.

- 회의 문서의 `Open` 항목이 설계 문서에서 확정처럼 쓰이지 않았는가?
- `후보`, `초안`, `권장` 표현이 DDL/API 계약에서는 hard constraint로 바뀌지 않았는가?
- 코드에 아직 없는 기술을 `현재 기술 스택`처럼 적지 않았는가?
- 레거시 코드 분석 결과를 새 구현 기준처럼 적지 않았는가?
- 배포, migration, rollback 순서가 운영 문서와 마이그레이션 문서에서 같은가?
- 보안 정책이 DB 설계, API 설계, 로그 정책에 동일하게 반영되었는가?
- OpenAPI, API 설계 문서, API 컨벤션의 endpoint 이름이 충돌하지 않는가?
- DB 컬럼 제약과 DDL 예시가 같은 말을 하고 있는가?
- LLM 컨텍스트가 최신 결정과 남은 질문을 압축해서 반영하는가?

충돌을 발견하면 우선순위는 다음과 같습니다.

```text
Accepted ADR
-> 명시적으로 확정된 설계 문서
-> 컨벤션
-> 회의 문서의 Proposed 후보
-> 레거시 분석 문서
```

회의 문서의 `Open` 항목은 구현 기준으로 사용하지 않습니다.

## LLM 컨텍스트 관리

- 구현 기준이 바뀌면 [LLM 컨텍스트](../llm-context.md)를 갱신합니다.
- LLM 컨텍스트에는 세부 설명보다 현재 기준과 금지사항을 압축해 적습니다.
- 회의 중인 내용은 확정된 기준처럼 쓰지 않습니다.
- 레거시 코드에서 유추한 내용은 source of truth가 아니라 참고라고 표시합니다.
